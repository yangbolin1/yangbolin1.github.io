## 🗺️ 破局：迷宫背后的四大阵营

别被那几十个 Java 文件吓到。剥去复杂的外衣，整个 Camera2 框架的 Java 层其实只在干四件事：

1. **管道工厂 (Session 家族)**：相机打开了，怎么把画面传输给 App？建管道。
2. **集装箱 (Metadata 家族)**：传输过程中，ISO、曝光参数怎么下发和上报？装集装箱。
3. **跨语言翻译官 (`marshal` 包)**：Java 的参数底层 C++ 不认识怎么办？找翻译官。
4. **特种部队 (高端工具箱)**：RAW图、慢动作、多摄平滑变焦等疑难杂症怎么处理？呼叫特种部队。

让我们一步步、充满逻辑地深入这四大阵营。

---

## 🏭 第一阵营：管道工厂 (Session 家族)

**🤔 小白疑问**：我都拿到 `CameraDevice` (相机实例) 了，为什么不能直接拍照，非要繁琐地 `createCaptureSession`？

**💡 核心逻辑**：
相机硬件就像一个水库，你的屏幕（`SurfaceView`）或者录像机（`MediaRecorder`）就像你家里的水龙头。水库里有水，水龙头也准备好了，但它们之间**没有连水管**！
**`Session` (会话) 就是这根水管。**

在源码目录中，你会看到不同规格的“水管”：

### 1. 基础自来水管：`CameraCaptureSession.java`
这是我们平时 90% 都在用的标准会话。
*   **作用**：它负责把你提供的画布（`Surfaces`）告诉底层 C++。底层会在内存中分配好 `GraphicBuffer`，将相机和画布连通。你调用的 `setRepeatingRequest()`（开启预览），就是把水泵打开，让画面源源不断地流过来。

### 2. 高铁专用管：`CameraConstrainedHighSpeedCaptureSession.java`
*   **痛点**：如果你要拍 **120fps 甚至 240fps 的慢动作视频**，标准水管会撑爆。因为在 240fps 下，每一帧只有 4 毫秒的时间，根本来不及让 Java 和 C++ 频繁地交互和通信。
*   **底层解法**：系统专门搞了这个高速会话类。在这个会话里，系统会**强制限制你的自由**（比如不让你随意切换对焦模式），并且强制开启 `BatchedOutputs`（批量模式）。
*   **大白话**：它不再一帧一帧地处理，而是让你一次性提交 8 帧的请求，底层闷头处理完，直接打包返回给你 8 帧画面。极致压榨性能！

### 3. 魔法滤镜管：`CameraExtensionSession.java`
*   **痛点**：微信自带的相机拍夜景，永远比不上系统自带相机（华为/小米）的“超级夜景”，因为系统相机的牛逼算法是不开源的！
*   **底层解法**：Android 引入了 Extension 机制。这个 Session 就像一个带有“魔法滤镜”的通道。当 App 通过它建立连接时，底层会把画面先绕道送给手机厂商（OEM）的私有算法库，处理完（提亮、降噪）再送回给 App。第三方 App 终于能白嫖原厂算法了！

## 📦 第二阵营：集装箱 (Metadata 家族)

建立好管道后，我们要怎么给相机下达命令？

**🤔 小白疑问**：相机有几百个参数（ISO、曝光补偿、人脸检测等），为什么源码里没有 `setIso()`、`setExposure()` 这样的具体方法？

**💡 核心逻辑**：
如果给每个参数写一个 `set` 方法，API 会爆炸，而且以后每增加一个新硬件特性，都要改 Android 源码！
所以，Google 设计了 **`Metadata` (元数据)** 机制。它就像一个 **Key-Value 键值对的集装箱**。

在源码目录中，这三个类是你绝对绕不开的“集装箱三剑客”，它们都继承自 `CameraMetadata.java`：

1. **`CameraCharacteristics` (产品说明书)**
   *   **方向**：底层 -> App（只读）
   *   **作用**：它装载的是这台手机**不可改变的物理属性**。比如：这颗镜头最大支持多少分辨率？有没有闪光灯？支持不支持光学防抖（OIS）？

2. **`CaptureRequest` (操作指令单)**
   *   **方向**：App -> 底层
   *   **作用**：每次拍照或预览，你都要填一张表。你在 Java 层调用 `builder.set(CaptureRequest.CONTROL_AF_MODE, ...)`，就是在这个集装箱里放进了一张“开启连续对焦”的纸条，然后顺着 Session 管道发给底层。

3. **`CaptureResult` (出厂质检报告)**
   *   **方向**：底层 -> App
   *   **作用**：底层拍完这一帧后，不仅会把画面给你，还会发回一份报告。里面记录了**这一瞬间真实的物理数据**：当前的真实 ISO 是多少？曝光时间具体精确到了多少纳秒？人脸在屏幕的坐标是哪里？

## 🗣️ 第三阵营：跨语言翻译官 (`marshal` 包)

这是子目录 `impl/marshal` 下的内容，极其隐蔽，但技术含量极高。

**🤔 小白疑问**：我们在 `CaptureRequest` 里设置裁剪区域，传的是一个 Java 的 `android.graphics.Rect` 对象。底层的 C++ 根本不认识 Java 对象啊，它怎么知道裁剪哪里？

**💡 核心逻辑**：
**跨越 Java 和 C/C++ 边界时，一切对象都必须粉碎成基础字节 (Byte)**。这就是 `marshal` (编解码/序列化) 包的使命。它就像联合国大会上的同声传译员。

在这个包里，你会看到几十个翻译官：
*   `MarshalQueryableRect.java`：它截获你的 Java `Rect` 对象，把里面的 `left, top, right, bottom` 抠出来，转成 4 个 32位整型（16字节）的内存块，顺着 Binder 发给 C++。
*   `MarshalQueryableEnum.java`：负责把 Java 的枚举，翻译成 C++ 的 `uint8_t` 标志位。
*   `MarshalQueryableString.java`：负责处理字符串的 C 语言结尾（补充 `\0`）。

> **🎓 架构师思维**：如果你以后需要做 NDK/JNI 开发，需要频繁在 Java 和 C++ 之间传递复杂结构体，这个 `marshal` 包的源码绝对是 Android 框架内最标准、最优雅的教科书！

## 🛠️ 第四阵营：特种部队 (高端玩家工具箱)

除了基础的流程，目录下还有几个零散但非常硬核的类，它们专门解决移动影像领域的疑难杂症。

### 1. 数字底片制造机：`DngCreator.java`
*   **概念普及**：平时我们拍的 JPEG 照片，是底层 ISP（图像处理器）经过了涂抹降噪、色彩渲染后的“精修图”。而 **RAW 格式 (DNG)**，则是传感器捕捉到的最原始、没有处理过的光子数据（拜耳阵列）。
*   **它的使命**：当你需要开发一款专业摄影 App 时，底层会把纯原始的 RAW 内存块扔上来。`DngCreator` 的作用就是按照 Adobe 的工业标准，把这块内存，连同色彩配置、黑电平数据等 Metadata，打包压缩成一张 `.dng` 后缀的文件，方便用户去 Lightroom 里无损调色。

### 2. 多摄融合管家：`MultiResolutionImageReader.java`
*   **痛点**：现代手机动辄 3-4 个镜头（超广角、主摄、长焦）。如果用户在录像时，手指从 0.5x 慢慢滑动放大到 5x，底层其实是在不同的物理镜头间疯狂切换。但问题来了：主摄是 5000 万像素，长焦只有 800 万像素。如果在录像途中，底层传回来的画面分辨率突然变了，App 渲染引擎直接就崩溃了！
*   **它的使命**：这个类是 Android 12 引入的特种部队。它在逻辑上把多个不同分辨率的物理摄像头，包装成了一个统一的虚拟流输出。它在幕后默默为你搞定动态分辨率分配和内存管理，让上层 App 以为自己一直在用同一个镜头平滑变焦。

## 🎯 全篇总结

看完这四个阵营，我们再回顾一下 Android Camera Framework 的 `camera2` 源码目录，你会发现一切都是那么井然有序：

1. **`Session` (管道)** 解决了数据流向哪里的问题，并演化出了高速公路和魔法滤镜。
2. **`Metadata` (集装箱)** 抛弃了臃肿的 API，用极致灵活的键值对解决了海量参数的下发与上报问题。
3. **`marshal` (翻译官)** 在幕后默默缝合了 Java 与 C++ 的语言鸿沟。
4. **特种工具类** 随时待命，拥抱计算摄影和多摄时代的新硬件挑战。
